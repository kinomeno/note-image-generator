# 開発経緯 — ととのえ

「ととのえ」がどんな思考の流れで現在の形になったか、の記録。
設計判断の「なぜ」を残し、今後の意思決定の土台にする。

---

## 1. 出発点：デスクトップEXE
最初に作ったのは Windows 向けデスクトップアプリ。
画像を note サイズに一括変換する実用ツールとして、ドラッグ＆ドロップ、プリセット、複数サイズ、テキスト追加、背景色、出力形式選択を実装。
その後さらに **リアルタイムプレビュー／フィルター／バッチ＋スケジュール／フラットデザインUI** まで盛り込み、高機能版へ。EXE化は PyInstaller。

→ 「全部入り」の強力版がデスクトップに完成。

## 2. Web版の構想
デスクトップが高機能化した反動で、**Web版はあえて真逆＝「誰でも迷わない、おしゃれでシンプル」** に振る方針へ。
当初のWeb版は単一サイズ変換＋ZIPだけの素朴な作りだったため、全面的に作り直すことに。

## 3. デザインの方向性決め
モックを作って比較。
- 3案（**ミニマル白＋インディゴ** / ウォーム / ミント）を提示 → **ミニマル**採用。
- さらに参考デザイン（白基調・余白多め・上品なタイポ・落ち着いたグレーUI）に寄せて再構成。
- 「ドラッグ＆ドロップ枠は、画像投入後にオリジナル↔変換後の比較ゾーンへ入れ替える」構成を採用。

## 4. プロダクトの芯を決めた要件
ユーザーとの対話で、Web版の性格が固まる：
- **1枚特化**（複数同時はやらない）。
- **ライブプレビュー**（設定変更＝即反映、を当然の挙動として実装）。
- **リセット**で初期化。
- **PCは縦スクロールを出さない／モバイルは横スクロールを出さない**。
- 無料＝note変換、**追加機能はパスワードで解放**（note有料記事にコードを載せる導線）。
  - パスワードは「調べられる人は使ってOK」というスタンス＝ソフトゲートで割り切り。

## 5. 命名「ととのえ」
アプリ名を **ととのえ** に決定。
画像を“ちょうどよく整える”という機能と、静か・上品な世界観を一語で表す。ロゴは明朝、語尾「え」に差し色。

## 6. 実装と検証
- Web版を単一HTMLで実装（Canvasで変換、Web Cryptoで解放判定、localStorageで状態記憶）。
- 接続中の実ブラウザ＋ヘッドレスChromeで自動・目視の両面テスト。
  - ハッシュ照合、1920×1006変換、背景色反映、UI遷移、単一DL/ZIP、解放、リセットを確認。
- 検証中に**オリジナルのサムネが壊れる不具合**（Blob URLの早期解放）を発見し修正。

## 7. 実使用フィードバックでの磨き込み
実際に触ってもらい、以下を反映：
- ヘッダーを細く・タイトル小さく。
- 解放済み表示は最小マークに（主張しすぎを是正）。
- 「即反映」表示は冗長なので削除。
- 出力サイズ等を **こまかい設定** に集約＋折り畳み＋開閉状態保存。
- **サイズスライダーが主役**になると判明 → preview直下へ大きく移動、無料機能化。
- プレビュー横のグレー余白が「完成画像の一部」と誤解されるため廃止し、**枠線で出力範囲を明示**。
- 鍵マーク（🔒）は世界観を壊すので不使用。
- 全体の文字が薄すぎたためコントラストを引き上げ。
- 開発・確認用に `?lock=1`（ロック状態プレビュー）を追加。

## 8. デザイン上の差し色
原色を避け、**くすみセージ `#5f8276`** を採用。白基調に上品に映り、「ととのえ」の静かな世界観に合う。

## 9. バージョン管理の開始
ここまでの完成度を区切りとして **v0.10** を起点にバージョン管理を開始。
プロジェクト内も用途別フォルダ（docs / desktop / design / ドキュメント）に整理。

---

## 設計判断の要点（まとめ）
- **引き算の設計**：無料画面は徹底的にシンプル。高度な操作は折り畳みの奥へ。
- **主役は1つ**：投入前はドロップ枠、投入後はサイズ調整。視線とマウス移動を最小化。
- **世界観優先**：色・余白・タイポ・アイコンの一貫性を、機能追加より優先して守る。
- **割り切った課金導線**：完璧な認証より、運用が軽いソフトゲート。
